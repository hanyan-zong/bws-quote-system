## 三、关键设计决策与反思

### 3.1 为什么 skill 放在 `balijob/skill/` 而不是 `预报价系统B端版本/skill/`?

**A**: 后者是项目级,只服务 BWS。前者是 balijob 工作目录级,覆盖所有项目 (爪哇报价 / 导游手册 / ERP / 自动化脚本)。**编程规范天然跨项目**,放低层不可达高层。

### 3.2 为什么不直接放 `C:\Users\001\.claude\skills\`?

**A**: Claude Code 的 skills 目录是给 `Skill tool` 自动加载用的,有 frontmatter 约束 (name 唯一性 / description 长度)。本 SKILL.md 是引用型文档,不是触发型 skill,放在数据目录引用更合适。如果未来想自动加载,再迁移 + 改 frontmatter。

### 3.3 为什么 GitHub 调研要写成"铁律"而不是"建议"?

**A**: 凭记忆写编程指南有两个隐藏成本:
- (1) 行业半年就会有新陈代谢 (callback → Promise → async/await,REST → GraphQL → tRPC),凭记忆改会写成"博物馆展品"
- (2) "我记得" 是不可审计的 — 没有 source URL,3 个月后无法判断哪条规则还有效

强制 WebFetch 实际仓库 README → 每条规则可溯源 → "更新历史"表格可审计 → 后续维护者(包括未来的我)能信任。

### 3.4 SKILL.md 600 行了,是否要拆?

**未拆。理由**:
- 单文件 Ctrl+F 比跨文件跳转快
- 600 行内一次 Read (2000 行默认) 能全装,不切片
- 拆了之后维护更新历史更难

**触发拆分的阈值**:
- 单文件 > 1500 行
- 出现明显独立可用的主题 (如"性能调优最高指南"独立成 ~500 行)

到了再拆,不预拆。(YAGNI 原则,本 skill 自己 1.3 节就这么说)

## 五、本日学到的元规则 (Meta-rules learned today)

1. **建立体系比解决单点问题更高 ROI** — 上午写 CLI 是单点,下午写 SKILL 是体系。后者管未来所有项目。
2. **铁律必须写进多处** — 单点容易遗忘,3 处冗余 (SKILL 内文 / 审计表 / feedback memory) 才能扛住时间。
3. **WebFetch 实际 README 比 WebSearch 摘要更可靠** — 摘要会丢清单,会过滤"次要"实战条目 (而那些往往是真正的金子)。
4. **frontmatter 的 `sources:` + `updated:` 是 skill 的体检报告** — 看一眼就知道这版还能不能信任。
