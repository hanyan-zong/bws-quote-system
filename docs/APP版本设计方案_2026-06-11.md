# BWS 预报价系统 APP 版本设计方案

> 2026-06-11 | 目标: 一套代码覆盖 Android / iOS / HarmonyOS NEXT
> 调研依据: GitHub/Gitee 实测 5 个仓库 + 4 组检索 (见文末来源)

## 一、选型决策 (先说结论)

**uni-app (Vue3 + TypeScript) + unibest 工程模板 + wot-design-uni 组件库**，一套代码编译出 Android APK、iOS IPA、鸿蒙 HAP，顺带免费得到 H5 和微信小程序版本。

### 候选对比

| 方案 | 鸿蒙 NEXT 支持 | 代码复用 | 团队匹配 | 结论 |
|------|--------------|---------|---------|------|
| **uni-app (Vue3)** | ✅ HBuilderX ≥4.27 官方支持, 编译为 ArkTS | ★★★ 现有前端就是 Vue3, 业务逻辑可直接迁移 | ★★★ | **选它** |
| Flutter (鸿蒙港版) | ⚠️ openharmony-sig 维护, 基线落后主线 (gitee 上 3.7.x-ohos), 模拟器只支持 Mac arm64 | ✗ 全套重写 Dart | ✗ 团队是 Python+Vue | 否 |
| Taro | ⚠️ 京东自家鸿蒙 APP 因兼容问题已退回原生开发 | △ React 体系 | ✗ | 否 |
| Capacitor/WebView 套壳 | ✗ 鸿蒙 NEXT 无 Android 兼容层, 套壳路线直接死 | ★★★ | — | 否 |
| 三端原生 | ✅ | ✗ 三套代码三倍成本 | ✗ | 否 |

**关键事实**:
- 鸿蒙 NEXT 已去掉 Android 兼容层 → "打个 APK 也能跑鸿蒙"的时代结束了, 必须编译为 ArkTS/HAP。
- uni-app 41.5k★, 明确支持"iOS、Android、鸿蒙、Web、各种小程序"; 鸿蒙端**只支持 Vue3** — 我们现有前端恰好是 Vue3, 没有 Vue2 历史包袱。
- 行业数据: 89% 的应用需要同时支持鸿蒙/安卓/iOS, uni-app 与 Flutter 是当前最成熟的两条路; 我们的技术栈决定了 uni-app 是唯一不换血的选择。

## 二、APP 架构总览

```
┌─────────────────────────────────────────────┐
│  uni-app (Vue3 + TS + Vite5)   一套源码      │
│  ├─ pages/      约定式路由                    │
│  ├─ stores/     pinia (用户/报价草稿/资源缓存)  │
│  ├─ api/        请求封装 (拦截器自动续 token)   │
│  ├─ components/ wot-design-uni + 自定义       │
│  └─ utils/      与 web 端共享的纯函数           │
└──────┬──────────┬──────────┬────────────────┘
       │ 编译      │          │
   Android APK  iOS IPA   鸿蒙 HAP (ArkTS)     (+H5/小程序 免费赠品)
       └──────────┴──────────┘
                  │ HTTPS + JWT
┌─────────────────▼───────────────────────────┐
│  FastAPI 后端 (现有, 不动存量)                 │
│  + /auth/token /auth/refresh  JWT 双 token   │
│  + 增量同步端点 (updated_at 游标)              │
│  + slowapi 限流 (login/refresh)              │
└─────────────────────────────────────────────┘
```

### 工程化 (采纳 unibest 模板, 2.8k★)

unibest = uni-app 社区公认的最佳实践集合, 直接拿来当骨架:
- **uniapp + Vue3 + TS + Vite5 + UnoCSS + wot-ui + z-paging**(长列表) + pinia
- 约定式路由 / layout 布局 / 请求封装 / 请求拦截 / 登录拦截 — 全部开箱即用
- Node ≥18 + pnpm; 鸿蒙发行: 项目导入 HBuilderX → 菜单发行到鸿蒙
- CI: GitHub Actions 跑 lint + 单测 + H5 构建; Android 走 HBuilderX cli 云打包; iOS/鸿蒙首发手动打包, 稳定后接云打包

## 三、后端增量设计 (只增不改, web 端零感知)

### 3.1 认证: JWT 双 token (移动端) 与 session cookie (web 端) 并存

现状是 session cookie + auth_gate 中间件, APP 端 WebView 外无 cookie 体系, 必须上 token:

```
POST /api/v1/auth/token    → {access_token (30min), refresh_token (14d)}
POST /api/v1/auth/refresh  → 旋转刷新: 旧 refresh 作废, 发新对 (防重放)
POST /api/v1/auth/logout   → refresh 入撤销表
```

- access token 短命 (30min), 只放 user_id + role, **不放任何敏感数据** (JWT 可被解码, 只是防篡改不是加密)
- refresh token 存服务端撤销表 (新表 `refresh_tokens`: jti / user_id / expires_at / revoked), 登出或异常即废
- APP 端存储: access 放内存, refresh 放系统安全存储 (iOS Keychain / Android Keystore / 鸿蒙 asset store, uni-app 有对应插件), **不落明文 storage**
- login/refresh 端点上 slowapi 限流, 防爆破
- auth_gate 中间件改为: cookie 有效 → 放行 (web 不变); 否则查 `Authorization: Bearer` → 放行 (APP)。十行级改动
- 现有 v0.9.2 角色 gate (super_admin/ops_manager/agent) 原样生效 — agent 在 APP 上同样看不到成本价

### 3.2 移动端专用增量

| 端点 | 用途 |
|------|------|
| `GET /api/v1/sync/resources?since=<updated_at>` | 资源价目增量拉取 (酒店/房型/季节价/车资), 支撑离线报价 |
| 现有 REST | 列表类补标准分页参数 (page/size), 响应加 total |
| `GET /api/v1/health` | 已有, APP 启动探活 + 版本检查 (version 字段已在) |

### 3.3 离线能力 (P2, 设计先行)

- 业务员在巴厘岛山区无网 → 本地缓存价目表 (storage + updated_at 游标增量刷新)
- 报价草稿存本地, 联网后提交; 冲突策略: 服务端 updated_at 优先 + 提示人工确认
- **不做**本地完整计价引擎副本 (DRY: 计价知识只在服务端一份, 离线只做草稿暂存, 联网才出正式价)

### 3.4 推送 (P2)

uni-app 原生集成 UniPush 2.0 (个推): 一个 SDK 覆盖安卓各厂商通道 + iOS APNs + 鸿蒙推送。场景: 报价单审批流转、节日价格档生效提醒 (`bws season suggest` 的成果直接变推送素材)。

## 四、功能分期 (小步增量, 每期可独立交付)

### P0 — MVP (核心闭环, 先让业务员手机上能报价)
1. 登录 (JWT) + 角色识别
2. 报价单列表 / 详情 / 状态流转 (z-paging 长列表)
3. **新建报价 case-based 表单** — 完整复刻 web 端 day_type 4 类 + 复制天 + 路线选择, 这是系统灵魂
4. 计价结果展示 (人均/总价/利润, agent 角色自动隐藏成本列)

### P1 — 管理能力上手机
5. 资源管理 (酒店/房型/季节价矩阵, 仅公司侧角色)
6. 季节日历查看 + 节日档一键生成 (调 season suggest 同款逻辑)
7. 报价单导出分享 (后端已有 CSV/xlsx 导出, APP 端调系统分享面板发微信)
8. AI 行程解析入口 (拍照/粘贴行程 → 解析 → 生成报价草稿)

### P2 — 体验差异化
9. 离线草稿 + 价目缓存
10. UniPush 推送
11. i18n (中/英/印尼语, wot-design-uni 自带 15 语言基建)

### 发布通道

| 平台 | 通道 | 前置 |
|------|------|------|
| Android | APK 直发/应用宝 | 无硬门槛 |
| iOS | TestFlight → App Store | Apple 开发者账号 $99/年, 审核 1-7 天 |
| 鸿蒙 | AppGallery (AGC) | 华为开发者实名认证 + 软著建议提前办 |

## 五、风险与对策

| 风险 | 对策 |
|------|------|
| 鸿蒙端 uni-app 插件生态弱于安卓/iOS | P0 只用核心 API (请求/存储/路由), 避开冷门插件; 每个三方插件先查鸿蒙兼容标 |
| iOS 审核拒 "纯 WebView 套壳" | uni-app 是原生渲染不是套壳, 但首版要有原生交互感 (下拉刷新/手势/系统分享) |
| Element Plus → wot-ui 重写工作量被低估 | 先做组件映射清单 + 1 个页面 (报价列表) 全链路打通再铺开 (曳光弹原则) |
| 双端 (web/APP) 业务逻辑漂移 | 计价/校验只在后端 (已是现状); 前端共享纯函数抽 `utils/` 同构包 |
| HBuilderX 云打包依赖 DCloud 服务 | 离线打包方案备份 (Android Studio / Xcode / DevEco Studio 本地工程) |
