## 一、项目背景与定位

### 1.1 业务问题
公司面向中国市场做印尼地接（巴厘岛 / 雅加达 / 瑟帕岛 / 龙目岛 / 科莫多）。当前痛点：

1. **B 端报价靠人工 Excel**：每来一个询单，业务员要手动汇总成本、计算汇率、加利润、写文档，每单 30-60 分钟
2. **资源数据散在多家供应商 Excel/PDF**：酒店报价、车队报价、餐厅菜单各自分散，更新不同步
3. **赌自费决策靠经验**：老业务员凭感觉判断"这单可以低于成本 500 抢"，新人完全不会
4. **客户拼出来的行程根本走不完**：销售/客户不熟巴厘地理，把跨南北景点排在一天，到现场才发现来不及
5. **报价不透明**：客户看不到清晰的拆分；管理者看不到团的真实利润结构

### 1.2 系统定位
**面向 B 端旅行社的智能预报价网页系统**，自动完成：
- 资源库统一维护（含 AI 文档采集）
- 行程组合（一日游模板 + 自由活动）
- 行程合理性自动校验
- 双币种计价（IDR 成本 → CNY 报价）
- 赌自费 AI 推荐
- 一键导出 Excel/PDF/Word 报价单

### 1.3 用户角色
- **管理员**：维护资源库、汇率、模板、规则；审核 AI 解析结果
- **B 端业务员**：选目的地 → 选行程 → 输出报价
- **B 端旅行社老板**（v0.2）：查看自己团队的报价历史与自费成单率

| 能力 | 实现技术 | 关键产出 |

| 资源库管理 | SQLAlchemy + FastAPI CRUD | `/resources/*` 全套 API |

| 文档导出 | openpyxl / WeasyPrint / python-docx | `/quotes/{id}/export` |

## 四、典型业务流程（端到端）

### 流程 A · 管理员录入新供应商酒店报价
1. 管理员上传 `JKT_HOTEL_2026.xlsx`
2. 后端 → openpyxl 抽 sheet/cell → Claude API 抽取 → 返回 12 家酒店 × 28 房型 JSON
3. 前端预览表格，管理员勾选/编辑/删除
4. 点 [确认入库] → 写 `hotels` + `hotel_rooms` 表
5. 修正记录写 `logs/ai_corrections.jsonl`

### 流程 B · 业务员生成 4 晚 5 天蜜月报价
1. 选目的地：巴厘岛（DPS）
2. 选日期：2026-08-12 ~ 2026-08-16，2 大人 0 儿童
3. 客户类型：蜜月
4. Day 1 抵达：酒店 The Mulia + 17 座车 + 中文导游 + 接机
5. Day 2 选模板"乌布文化一日游"（系统自动锁定景点/餐厅范围）
6. Day 3 标记自由活动
7. Day 4 标记自由活动
8. Day 5 离开：早餐 + 海神庙 + 送机
9. 点 [计算] → 后端依次执行：
   - Pricing Engine 汇总 IDR 成本（按汇率 2300 换 CNY）
   - Feasibility Engine 校验所有天数的可行性
   - 假设 Day 5 海神庙 + 送机超时 → 弹出警告 + 建议改 Day 5 早晨海神庙 → 应用建议
   - Gamble Engine 看到 2 天自由活动（占比 40%）+ 蜜月 + 旺季 → 推荐 ¥450 让利
10. 业务员点 [接受推荐] → 系统更新对外报价
11. 点 [导出 Excel] → 客户拿到的是仅显示 RMB 总价的报价单

### 流程 C · 团队回程后回写自费收益
1. 财务进入 `/quotes/{id}` 页
2. 填实际自费收入（如 ¥1850/团）+ 总利润
3. 系统更新 `gamble_history`
4. 每周一脚本聚合 → 更新 `optional_tours.historical_purchase_rate`
5. 下次同类型报价的赌额推荐更准

## 八、关键风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| AI 解析准确率不达标 | 管理员仍要大量手改 | 用户修正写日志，每周分析改 prompt |
| 距离矩阵冷启动数据不全 | 校验经常报"缺数据" | 先用母库车程矩阵 + 兜底默认值 + Google API 回填 |
| 赌自费推荐失准 | 团亏损或抢单失败 | 风险护栏（最大亏损 25%）+ 人工最终决定 |
| Claude API 不稳定 | 用户体验差 | mock 降级 + 重试 + 友好提示 |
| 多业务员并发改资源 | 数据冲突 | v0.2 引入乐观锁 + 修改记录 |
