## 一、修的真 bug:`bws.exe` 入口 `ModuleNotFoundError`

### 现象

```
$ ./bws.bat --help
Traceback (most recent call last):
  File ".venv/Scripts/bws.exe/__main__.py", line 2, in <module>
ModuleNotFoundError: No module named 'app'
```

### 根因

`.venv/Lib/site-packages/` 残留一个 **`~ws_quote-0.8.4.dist-info`** 目录(波浪线前缀 = pip 半卸载状态)。

时间线推断:
1. 2026-05-16 v0.9 发版前,venv 装的是 `bws-quote 0.8.4`(editable)
2. 改 `pyproject.toml` version 到 `0.9.0` 后,没有 `pip install -e .` 重装
3. 后续(可能 0.9.0 → 0.9.3 又改了版本号)再装时 pip 半卸载老版本,留下 `~ws_quote-0.8.4.dist-info` 残骸
4. `pip show bws-quote` 此时报 `Package(s) not found`(因为 dist-info 名字是 `~ws_quote` 不是 `bws_quote`,pip 索引断了)
5. 但 `.venv/Scripts/bws.exe` 还是 0.8.4 留下的 entry-point launcher;它的 shebang 找 `app.cli:main`,而 `app` 模块在 backend/,需要 pip 把 editable 路径写进 site-packages 才能 importable;残缺状态下找不到 → `ModuleNotFoundError`

### 修法

```powershell
.venv\Scripts\python.exe -m pip install -e .   # 重新注册 editable 包
Remove-Item ".venv\Lib\site-packages\~ws_quote-0.8.4.dist-info" -Recurse -Force
```

之后:
- `pip show bws-quote` → `Version: 0.9.3` ✅
- `./bws.bat --help` → 正常输出 ✅
- `bws.exe` 入口指向当前 `app.cli:main` ✅

### 这条要进 [[project_bws_cli]] memory

**升级 `pyproject.toml` 的 `version` 后必须 `.venv\Scripts\pip install -e .` 重装一次**,
否则 entry-point 还指向旧 dist-info,而旧 dist-info 在 pip 第二次 install 时会被半卸载留下 `~` 残骸,导致 `bws.exe` 跑不通且 `pip show` 报 not found。

## 二、权限白名单建设:从"每次询问"到"95% 静默"

### 第一版(只读白名单 · 12 条)

跑 `fewer-permission-prompts` skill,扫了最近 15 个会话 transcripts,统计后发现:
- 高频调用大头是 `cd`/`ls`/`cat`/`grep`/`find`/`echo` + `git status/log/diff` ——
  这些 Claude Code 默认就**自动放行**,根本没有"询问"。不在白名单也不会烦你。
- 真正每次都问的是: `pytest`(49 次)、`curl`(28)、`uvicorn`(20)、`mkdir`(14)、`pip install`(7)、`alembic`(9)、`bws.bat`(8)、`rm`/`cp`/`mv`(多次)
- 其中 **49 次 pytest / 8 次 bws.bat list show / 2 次 alembic current** 等是只读的,可以放心加白

第一版纯只读 12 条:

```json
"Bash(pytest *)",
"Bash(.venv/Scripts/python.exe -m pytest *)",
"Bash(.venv/Scripts/python.exe -m pip show *)",
"Bash(.venv/Scripts/python.exe -m alembic current)",
"Bash(.venv/Scripts/python.exe -m alembic history)",
"Bash(.venv/Scripts/python.exe --version)",
"Bash(./bws.bat --help)",
"Bash(./bws.bat quote list *)",
"Bash(./bws.bat quote show *)",
"Bash(./bws.bat db stats)",
"Bash(./bws.bat server status)",
"Bash(where *)"
```

但这条覆盖不到写操作 — user 说 95%+ 都不要问。

### 第二版(宽松白名单 + deny 兜底)

策略转换:
- `defaultMode: "acceptEdits"` — Edit/Write 文件改动自动通过
- `allow` 扩到 ~60 条,覆盖**所有日常 BWS 开发工作流**(写操作也放):
  - venv 全套可执行 `.venv/Scripts/{python,pip,bws,uvicorn,alembic}.exe *`
  - `./bws.bat *` 通配(整个 CLI 任意子命令)
  - git 写: `add` / `commit` / `push` / `fetch` / `pull` / `checkout` / `merge` / `rebase` / `stash` / `restore` / `switch` / `tag` / `remote`
  - 文件: `mkdir` / `cp` / `mv` / `touch` / `chmod` / `rm` / `rmdir`
  - 网络: `curl` 限本机 `localhost` / `127.0.0.1`
  - GitHub: `gh pr/issue/repo/release` 创建编辑
  - Docker: `compose` / `logs` / `inspect` / `start` / `stop` / `restart` / `build`
  - 前端: `npm` / `node` / `npx`
- `deny` 列表保留 catastrophic 拦截(20+ 条):
  - `rm -rf /` / `rm -rf C:/` / `rm -rf ~` 系列
  - `git push --force` / `-f`(任何形式强推)
  - `git reset --hard *`
  - `git clean -fd` / `-fdx`(强清未追踪)
  - `git branch -D *`(强删分支)
  - `dd if=*` / `mkfs *` / fork bomb

### 落地位置

`预报价系统B端版本/.claude/settings.json`(project-level,跟随仓库)
- **不是** `~/.claude/settings.json`(那是全局)
- **不是** `.claude/settings.local.json`(那是 gitignored,不跟仓库)

### 剩余 5% 仍被询问

- 全新没见过的命令(防止恶意工具调用乱跑)
- 命中 deny 列表的灾难性操作
- 非本机的 curl/网络请求(防数据外泄)
- 写到项目外的文件(系统文件、其他项目)

## 四、踩坑/反思

### 坑 1:用户首次说 "bwscli" 我没立即跑命令验证

memory 里写着 v0.9.0 + "本仓库已推 GitHub",我差点直接照念。**幸亏**先跑了 `./bws.bat --help`,才发现入口已坏。

→ 强化 [[feedback_no_fantasy_reflect_each_step]]:**memory 是 5 天前的快照,不是 live 状态**;涉及具体命令/版本号时,**实测 > 念稿**。

### 坑 2:第一版白名单太保守,用户要求二次扩大

`fewer-permission-prompts` skill 的口径是"read-only only"。但用户实际诉求是"95% 不问",这跨过了 skill 的边界。第二版主动加 `acceptEdits` + 写操作白名单 + deny 兜底,属于 skill 外的扩展。

→ skill 是工具不是教条,**user 诉求超 skill 边界时,要主动跨过去**,不要被 skill 的 "Do not add to deny" 等内置限制锁死(那条限制是 skill 写给"自动模式"用的,user 明确要求的场景例外)。

### 坑 3:scan 脚本第一版用 heredoc 写 Python,反斜杠在 bash heredoc 里被吃掉

`r'[\s/\]'` 变成 `r'[\s/]'`,正则 `unterminated character set` 报错。

→ **Python 脚本不要用 bash heredoc 写**,直接 Write tool 写到 /tmp/ 路径,再 python 跑。这条加进 [[feedback_windows_python_env]] 备查。
