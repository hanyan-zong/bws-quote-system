## 为什么要做

- 昨天我做完"普通用户看不到管理 tab"的 UI 改动后,跟你提过:**前端只是把入口藏起来了,真正懂技术的业务员还是可以绕过浏览器直接调接口改资源**。
- 你今天确认要做后端权限,这一改就是把这扇后门关上。

举个具体例子:在 v0.9.1 之前,一个业务员账号即使在浏览器上看不到"设置"页,他完全可以打开命令行:
```
curl -X PUT http://你的服务器/api/v1/settings/exchange-rate -d '{"rate_cny_to_idr": 100}'
```
这条命令会把汇率改成 1 RMB = 100 IDR,然后他用这个汇率做完一单超低价报价,你都不知道。

v0.9.2 之后,同样这条命令会收到 **HTTP 403 无权**,改不了。

## 风险 (实事求是)

- **已知小问题**:无。59 个自动检查全过,包括 15 个专门验证"业务员被拒"的新测试。
- **要观察一段时间的**:**没观察到**。
- **明确没做的**:
  - **GET (读) 接口没加权限校验**。业务员能看到 IDR 成本字段?**不能** — 我们 v0.4 起就有字段过滤(`filter_resource_list`),业务员的 GET 响应里 `cost_idr_*` 这些敏感字段会被去掉。
  - **报价相关接口没动**(`/quotes/*`)。业务员本来就应该能创建/计算/导出自己的报价,这是核心业务流程,不动。
  - **AI 文档采集接口没动**(`/ai-parser/*`)。已知缺口 — 业务员理论上可以调 AI 入库资源。下次开工要把 `/ai-parser/extract` 和 `/ai-parser/confirm` 也加上 admin gate。

### 设计

权限框架在 v0.7 已经写好 (`feature_permissions.require_feature` + `utils/permissions.require_role`),但 routers 一个都没接进去 — 这是个**遗留漏洞**,v0.9.2 把 router 这一层接上。

用 FastAPI 的 `dependencies=[Depends(...)]` 列表注入,不动函数签名,改动最小。
