# 04 · 赌自费算法

> 版本 v0.2 | 2026-05-05
> 解决问题：行程含自由活动时，敢以多少钱低于成本报价以抢单

---

## v0.2 增量（重要）

在 v0.1 基础上新增三大核心增强：

### 1. 用户可自定义"不赌自费规则" (NoGambleRule 表)
- 表 `no_gamble_rules` 存储规则
- 一条规则 = `name` + `description` + 多个 `conditions` (JSON 数组) + `active` + `priority`
- 引擎按 priority 降序评估；任一规则的所有 conditions 全部命中 → 系统判定不赌、返回原因
- 支持的 condition.type：
  - `customer_type_in` — 客户类型 ∈ value
  - `free_hours_lt` / `free_hours_gt` — 总自由小时数比较
  - `total_days_lt` / `total_days_gt` — 总天数比较
  - `pax_total_lt` / `pax_total_gt` — 总人数比较
  - `is_first_time_agency` — 首次合作 B 端
  - `all_meals_included` — 全程含餐
  - `spa_already_booked` — 已含 SPA
  - `water_already_booked` — 已含水上项目
  - `season_in` — 季节属于 value 列表
- 默认 5 条规则（seed）：
  1. 自由时间不足 4h 不赌（priority 100）
  2. MICE/婚礼短行程不赌（priority 90）
  3. 全程含餐 + 已含 SPA 不赌（priority 80）
  4. 首次合作 + 小团不赌（priority 70）
  5. 老年团淡季不赌（priority 60，默认 inactive）

### 2. 自由时间细粒度 (free_hours)
- `QuoteDay.free_hours` 字段：0=全程行程 / 4=半天自由 / 8=全天自由
- 算法用 `free_hours_total / (total_days × 8)` 算自由比，比"整天计数"精准
- 4 晚 6 天行程含半天自由 → free_hours_total=4，free_ratio=4/48=8.3%

### 3. 自费项目重叠剔除 (category + overlap_*_ids)
- `OptionalTour.category` 取值：`spa / sunset / sunrise / food_upgrade / performance / water / shopping / temple / shows / island_trip`
- 显式重叠字段（JSON 数组）：`overlap_attraction_ids` / `overlap_restaurant_ids` / `overlap_spa_ids` / `overlap_water_ids`
- 引擎自动检测：
  - `category=spa` 且行程已设 `spa_id` → 排除该自费
  - `category=water` 且行程已设 `water_activity_id` → 排除
  - `category=food_upgrade` 且 `all_meals_included=True` → 排除
  - 显式 ID 重叠 → 排除
- 排除原因写入 `excluded_optional_tours` 数组返回前端

### 4. 兜底自动判定不赌
- `free_hours_total = 0` → 自动不赌（无自由时间无机会卖自费）
- 所有自费项目都被覆盖 → 自动不赌

## 一、业务背景

地接团的真实利润 = 行程明面利润 + 自费项目转化收入

举例：4 晚 5 天行程：
- 明面成本：¥3,800/人
- 行程含 2 天自由活动
- 这 2 天客人大概率买自费（海豚团/Spa/包车去网红店）
- 历史数据：人均自费消费 ¥600，毛利率 50% → 我方毛利 ¥300

→ 这种结构可以**先以 ¥3,500/人报价（亏 ¥300）抢单**，赌自费补回来还能赚 ¥0~¥100。

→ 这就是"赌自费"。算法目标：让 AI 帮你判断敢赌多少。

## 三、计算流程

### Step A · 自费购买能力评估
```
expected_optional_revenue_per_pax (CNY)
  = Σ over each available_optional_tour:
     (sale_price_cny × historical_purchase_rate × audience_match_factor)
```

`audience_match_factor`：
- 自费的 `target_audience` 与 `customer_type` 匹配 → 1.0
- 部分匹配 → 0.6
- 不匹配 → 0.2

### Step B · 自由时间转化系数
```
free_time_factor = min(free_days / total_days × 1.5, 1.0)
```
意思：自由活动占比越大，转化越高（但不无限放大，封顶 1.0）。

### Step C · 客户类型加成
| customer_type | 加成 |
|---------------|------|
| 蜜月 | 1.30 (Spa+海豚团高频购买) |
| 亲子 | 1.15 (儿童套餐+蓝梦岛家庭团) |
| 年轻人 | 1.20 (网红打卡+水上项目) |
| 家庭 | 1.00 |
| 老年 | 0.65 (低消费倾向) |
| MICE | 0.80 (集中行程为主) |
| 婚礼 | 0.95 (主活动占满，自费机会少) |

### Step D · 季节加成
| season | 加成 |
|--------|------|
| high | 1.10 |
| shoulder | 1.00 |
| low | 0.90 |

### Step E · 期望自费毛利
```
expected_optional_profit_per_pax
  = expected_optional_revenue_per_pax
    × average_margin_rate         (默认 0.5，可在 settings 调)
    × free_time_factor
    × customer_factor
    × season_factor
```

### Step F · 推荐赌额（中位数）
```
recommended_gamble_cny = expected_optional_profit_per_pax × safety_ratio
```
`safety_ratio` 默认 0.7（只赌期望毛利的 70%，留 30% 风险垫）。

### Step G · 区间下限/上限
```
low_bound  = recommended × 0.6
high_bound = recommended × 1.2
```
低值=保守报价，高值=激进抢单。

### Step H · AI 信心分

把以上所有输入 + 历史成功率统计 + 目前 `gamble_history` 数据扔给 Claude，让它给出一个 0~100 的信心分，并返回一段自然语言解释：

```
"基于该 B 端旅行社过往 12 单（命中 9 / 未中 3）+ 蜜月客群 + 旺季 +
2 天自由活动（占比 40%），AI 评估推荐赌额 ¥520/人，区间 ¥310~¥620。
信心分 78%。主要风险：旺季 Spa 涨价可能压缩自费毛利。"
```

## 六、成单后回写 → 反哺模型

成团结束后，财务把实际自费收入填入：
```sql
UPDATE gamble_history
SET optional_tours_revenue_cny = 1850.00,
    profit_actual_cny = 4200.00,
    won_or_lost = 'won'
WHERE quote_id = 1234;
```

每周脚本聚合：
- 不同 customer_type × season × free_ratio 的实际购买率
- 更新 `optional_tours.historical_purchase_rate`
- 更新 `audience_match_factor` 矩阵
- 输出 `logs/gamble_weekly_report.md`

3 个月后切换到"AI 直接训练 small predictor"模式（用 sklearn 或 XGBoost 在历史数据上训一个回归模型，预测期望自费毛利）。

## 七、风险护栏（必须的）

无论 AI 给多激进的建议，强制以下硬约束：

1. **赌额上限**：不超过明面成本的 25%（防止亏太狠）
2. **儿童不算**：12 岁以下客人不计入自费转化（他们家长才买）
3. **首次合作**：与首次合作的 B 端旅行社，赌额自动 ×0.5
4. **MICE/婚礼**：固定行程为主，自由活动少 → 强制赌额上限 ¥150
5. **管理员可全局关闭**：`settings.enable_gambling = false` 时整套机制不生效
