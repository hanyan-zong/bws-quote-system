## 一、🔴 P0 alembic 真正成为 schema 单一来源

### 当前状态(双轨)

```
新 DB 创建路径:
  bws db init     → Base.metadata.create_all (拿 ORM 反推的当前 schema)
  bws db migrate  → alembic upgrade head (跑空 baseline,只写 alembic_version)

已存在 DB 升级路径:
  ORM 加新字段 → alembic revision --autogenerate → bws db migrate
```

`init_db()` 用 `create_all` 建初始表,alembic 只管"未来变更"。这是符合 SQLAlchemy + alembic standard practice 的"半截方案",但**不是 single source of truth**:

- 新 DB 的初始 schema 由 ORM 模型隐式决定,**没有版本号、没有 migration history、回滚不了**
- 如果未来要"看 v1.0 schema 是什么样",只能 checkout v1.0 的代码 + 跑 init_db,看不到迁移历史

### 目标(单源)

```
新 DB:  仅 alembic upgrade head 即可拿到完整 schema
老 DB:  init_db 探测无 alembic_version 表时自动 stamp head
ORM:    仅作为 metadata 的载体,不直接驱动建表
```

### 实施步骤(估 1-2h)

1. **autogenerate 完整 0001 反映 v0.8.x schema**
   ```powershell
   cd backend
   # 清空 tmp DB 让 autogenerate 从空状态比对当前 ORM
   $env:BWS_DATABASE_URL = "sqlite:///../_tmp_for_autogen.db"
   New-Item -ItemType File ../_tmp_for_autogen.db -Force
   ..\.venv\Scripts\python.exe -m alembic revision --autogenerate -m "schema_v0_8_4_full"
   # 检查 migrations/versions/0001_schema_v0_8_4_full.py
   ```
   - **核对假阳性**:sqlite autogenerate 对默认值、Decimal precision、index、约束 容易误报。逐表核对一遍(约 20 个 ORM 模型)
   - 删 ALTER 痕迹:之前 v0.4-v0.8 的列在 ORM 里都有 default/nullable 信息,autogenerate 应能正确反映

2. **改 `app/database.init_db()` 为 alembic-driven**
   ```python
   def init_db() -> None:
       from alembic import command
       from alembic.config import Config
       from .config import BACKEND_DIR
       from sqlalchemy import inspect

       insp = inspect(engine)
       has_alembic = "alembic_version" in insp.get_table_names()
       has_quotes = "quotes" in insp.get_table_names()

       cfg = Config(str(BACKEND_DIR / "alembic.ini"))
       cfg.set_main_option("script_location", str(BACKEND_DIR / "migrations"))

       if has_quotes and not has_alembic:
           # 老 DB (已经有数据但从未接入 alembic) → 标记到 head, 不重跑 0001
           command.stamp(cfg, "head")
       else:
           # 新 DB 或已接入 alembic 的 DB → 正常升级
           command.upgrade(cfg, "head")

       _ensure_bootstrap_admin()
   ```

3. **删 `Base.metadata.create_all`** — 此时 alembic 是唯一 schema 来源

4. **测试更新**
   - `test_db_init_no_seed_creates_tables`:验证 tmp DB(空)→ init → 直接走 alembic upgrade → 完整 schema 建出
   - 加一个新测试:模拟"老 DB"场景 — 先用 SQLAlchemy `create_all` 建表 + 不建 alembic_version → init → 应当 stamp head(而非 upgrade)→ 验证 alembic_version 写入 + 没新建表

5. **生产 DB 一次性**:可选,如果用户已经手动跑过 `alembic stamp 0000_baseline` 则不需要;新 init_db 也会自动探测处理

### 风险

| 风险 | 缓解 |
|---|---|
| autogenerate 假阳性导致 0001 跟实际 schema 不一致 | 逐表人工核对(~20 模型 × 平均 8 字段 = 160 项),用 `sqlite3 .schema` vs autogenerated 文件对照 |
| 老 DB 探测逻辑误判(比如有 `quotes` 表但实际是测试残留) | 用更严格的探测:`alembic_version` 存在 → upgrade;`alembic_version` 不存在但有任何 ORM 表 → stamp head;**完全空** → upgrade (从 0001 建表) |
| stamp head 跳过了某些迁移 → 老 DB 缺特定列 | 当前用户 DB 已在 v0.8.4,所有列齐;但写 README 警告"v0.7 → v0.8.4 间老用户必须先用旧 init_db 跑一次升完,再切到新版" |
| 测试套件中 subprocess 跑 init 多了一次 alembic upgrade,慢 1-2 秒 | 接受 |

### 决策点

**做这个之前必须确认**:用户的生产 DB 是否已经手动跑过 `alembic stamp 0000_baseline`?
- 已跑:新 init_db 上线后会走 upgrade 0001(从 0000 升到 0001),需要 0001 在已有 schema 上是 noop(autogenerate 反映的就是当前 schema,理论上 noop;但 batch_op 可能会重建表)→ **危险**
- 未跑:新 init_db 上线后探测无 alembic_version → 自动 stamp head → 安全

**结论**:这个重构要么趁早做(生产 DB 还没 stamp 之前),要么写明"必须先 unstamp/重置"。**先和用户沟通生产 DB 状态再开工**。

## 二、🟡 P1 把 v0.4-v0.8 的 ALTER 历史迁成真实 migration 序列

P0 完成后,0001 是个"完整 schema snapshot"。如果想要**完整 migration history**(v0.4 → v0.5 → ... → v0.8.4 的真实演进),可以:

1. 重写 0001 为"v0.3 baseline"(只含 v0.3 时的列)
2. 加 0002 v0.4 多用户(`quotes.created_by_user_id` / `agency_id`)
3. 加 0003 v0.5 赌自费回写(`gamble_history.strategy_id` 等)
4. ...直到 0007 v0.8.4

**收益**:能 downgrade 到任意历史版本(回滚 / 历史调查 / 复制 v0.5 schema 给 demo)  
**代价**:6-7 个 migration 文件 + 每个都要测试,工期 3-4h  
**判断**:除非真有"回滚到 v0.5 demo"的场景,否则**不必做**。P0 单源已足够。
