## 一句话总结

报价系统终于知道**圣诞和淡季不是一个价**了。

---

## 之前 vs. 现在

### 之前(问题)

- 一个酒店一年只有**两个价**:`旺季价` 和 `淡季价`,二选一。
- 报价单选 `季节=旺季` → 全年所有日期都按"旺季价"算,**不区分 8 月暑假 / 春节 / 圣诞 / 元旦**。
- 政府税(21%)、服务费(10%)、印尼旅游税(IDR 150,000/人)**完全没在系统里**,业务员每次手工加。
- 圣诞夜酒店强制收的 Gala Dinner 包(几百万 IDR/晚)**完全没建模**,业务员忘记加 → 直接亏。

### 现在(解决)

- 每个房型可以挂 **5 档季节价**: 淡季 / 平季 / 旺季 / 高峰 / 节日。
- 系统有一张**全局日历表**:告诉系统 "12/20–01/05 是节日档"、"7-8 月是高峰"、"4-5 月是淡季"。
- 算价时:**先看日期 → 自动判断季节档 → 自动取该档价格**。
- **附加费一次配置,永久叠加**:政府税 21% 可以挂"全局生效",所有酒店所有日期自动加 21%;旅游税 IDR 150k/人/晚 同理。
- **节日捆绑包独立建模**:圣诞 Gala 的 IDR 280 万/房 + 50 万/人 在系统里,一旦旅客住到 12/24,系统自动把这笔加进去 + 把当晚另设的晚餐自动跳过(避免双算)。

## 业务员实际怎么用

### 场景 1:录入 The Mulia 4 档季节房价

打开 `🎄 季节定价` 新 tab → 不,先去 `🗂 资源库` 老 tab 建酒店(只填一次基本信息)→ 然后去 `🎄 季节定价` 给该房型挂 4 档房价(淡/平/旺/节日)。

> 当前 v0.9.4 房型多档录入用 API,UI 暂从"附加费/节日包/季节日历"3 个角度做;房型多档的可视化录入下一版补。**但 AI 采集已经支持**——把酒店合约直接喂给 AI,4 档价格能自动入库。

### 场景 2:配 2026 圣诞新年旺季日历

`🎄 季节定价 → 📅 季节日历` → `+ 新增日历区间`:

| 字段 | 填什么 |
|------|--------|
| 名称 | 2026 圣诞新年 |
| 季节档 | 节日 |
| 起始 | 2026-12-20 |
| 结束 | 2027-01-05 |
| 优先级 | 10(数字越大越优先,日期重叠时取大的)|

保存。从此**所有报价单上 12/20–01/05 这段的房费都按"节日档"取价**,不需要业务员手动改 `季节=旺季`。

### 场景 3:配印尼政府税 21% + 旅游税 150k

`🎄 季节定价 → 💰 附加费` → `+ 新增附加费`:

**政府税:**
- 名称 = `Government Tax 21%`
- 类型 = 政府税
- 计算方式 = 百分比 (在房费上叠加)
- 金额 = 21
- 适用酒店 = (留空,全局)
- 启用 = ✓

**旅游税:**
- 名称 = `Indonesia Tourist Tax`
- 类型 = 旅游税
- 计算方式 = IDR/人/晚
- 金额 = 150000
- 适用酒店 = (留空,全局)

从此**每张报价单的每一晚自动 +21% + 150k×人数**,业务员永远不会忘。

### 场景 4:配 The Mulia 圣诞 Gala

`🎄 季节定价 → 🎄 节日捆绑包` → `+ 新增`:

| 字段 | 填什么 |
|------|--------|
| 酒店 | The Mulia |
| 名称 | Christmas Eve Gala Dinner |
| 有效起 | 2026-12-24 |
| 有效止 | 2026-12-24 |
| 强制叠加 | ✓ |
| 房费 IDR | 2800000 |
| 人头费 IDR | 500000 |
| 包含 | Gala Dinner + 烟花 + 红酒 1 瓶 |
| **替换当晚晚餐** | ✓ ← 关键!避免双算 |

从此**12/24 住 The Mulia 的报价单**自动加 2.8M/房 + 500k×人,且当晚另设的晚餐被跳过。

## AI 文档采集升级

合约 PDF / Excel 里写"圣诞期间 +80% / 春节 +120%"这类多档季节价,AI 现在**直接识别并拆成多条 `room_rate`**入库,而不是塞进老的 low/high 两档。

合约里写"21% Gov Tax + 10% Service + IDR 150k/人/晚 Tourist Tax",AI 直接抽 3 条 `surcharge` 入库。

合约里写"Christmas Eve Gala IDR 2.8M + 500k/pax, mandatory, includes wine + fireworks",AI 直接抽 1 条 `hotel_package` 入库,`mandatory=true` + `replaces_dinner=true` 都自动标。

---

## 与老数据的兼容性

- 老的 `cost_idr_low` / `cost_idr_high` 字段**保留不动**。
- 如果一个房型**没配新的 5 档**,系统自动 fallback 到老的两档行为(low→淡/平,high→旺/高峰/节日)。
- 老的 67 个测试**全部通过**,新加 14 个测试(单元 10 + HTTP 4),**共 81 PASS**。

## v0.9.4.1 追加 (2026-05-24 当日):学校假期/国庆附加费

`charge_type` 增加 3 个专用类型,让业务员直接选,不用每次手填名字:

| 类型 | 选项名(UI) | 典型用法 |
|------|-----------|---------|
| `summer_break` | 🏖 暑期附加 (7-8 月) | 配 `valid_from=2026-07-01 / valid_to=2026-08-31` |
| `winter_break` | ❄️ 寒假附加 (1-2 月) | 配 `valid_from=2027-01-20 / valid_to=2027-02-20` |
| `national_day_break` | 🇨🇳 国庆附加 (10/1-10/7) | 配 `valid_from=2026-10-01 / valid_to=2026-10-07` |

**关键**:这 3 类必须配 `valid_from/to`,否则会全年误算。AI prompt 已写入这条规则。

### 端到端验证(新增 5 个测试,全 PASS)

真实建数据库,让报价系统按日期自动选档 + 叠加附加费:

| 场景 | 日期 | 房价档 | 政府税 21% | 假期附加 | **总计 IDR** |
|------|------|--------|-----------|---------|-------------|
| 🏖 暑期 | 2026-07-15 | peak 4M | 840k | 暑期 200k×2 = 400k | **5,240,000** |
| 🇨🇳 国庆 | 2026-10-03 | peak 4M | 840k | 国庆 20%×4M = 800k | **5,640,000** |
| ❄️ 寒假 | 2027-02-01 | peak 4M | 840k | 寒假 150k×2 = 300k | **5,140,000** |
| 普通平日 | 2026-03-15 | shoulder 2M | 420k | — | **2,420,000** |
| 边界 | 2026-08-31 | peak 4M | 840k | 暑期 400k(最后一天仍触发)| **5,240,000** |
| 边界 | 2026-09-01 | shoulder 2M | 420k | — (next day 不触发)| **2,420,000** |

**结论**:日期 → 季节档 → 房价 → 假期附加 全链路自动,**业务员只需建好数据,后续报价 0 心智负担**。
