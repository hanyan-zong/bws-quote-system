## 三、🟡 P1 — 体系性增强

### 3.1 SKILL 拓展边界 (是否要建新的"最高指南"?)

当前 `coding-and-testing-standards` 是杂糅。深度领域可能值得独立:

| 候选 skill | 触发场景 | 主要参考 |
|----------|---------|---------|
| `api-design-standards` | 设计 REST / GraphQL / RPC 接口前 | shieldfy/API-Security-Checklist, microsoft/api-guidelines |
| `data-engineering-standards` | 写 ETL / 数据仓库 / 报表 | dbt-labs/best-practices, data-engineering-handbook |
| `frontend-standards` | 写 Vue / React 组件 | airbnb/javascript, vuejs/style-guide |
| `incident-response` | 出生产故障时怎么走 | google/sre-book PR / postmortem 章节 |

**判断标准**: 当前 SKILL 某一节超过 200 行 + 有独立信息源 + 触发场景明确 → 拆出来。**当前都不到,暂不拆。**

### 3.2 怎么让"五步工作流"真的被执行?

**问题**: 工作流写在 SKILL 第 7 节,但人 / Claude 都容易跳过"侦察"和"复盘"。

**机制改进**:
- [ ] 用 `update-config` skill 配置 Claude Code 的 `UserPromptSubmit` 或 `SessionStart` hook,自动 Read coding-and-testing-standards SKILL.md
- [ ] 或者用 `/loop` 自定义 slash command,在写代码任务里强制走五步
- [ ] 或者每个项目 CLAUDE.md 顶上加一行 "在本项目修改任何代码前先读 `skill/coding-and-testing-standards/SKILL.md` 五步工作流"

**优先 hook 方案** — 自动化最少摩擦。

## 五、🔵 元反思 — 今天暴露的反复问题

### 5.1 "P0 越堆越多" 现象

5/11 立的 P0 (E2E / 行结尾规范 / GitHub push dry-run / 赌自费重设计) 三天后 (5/14) **没一项动**。今天又新加了 P0 (CLI 测试 / 用上 SKILL)。

**根因**: 用户每次开工都给新需求,新需求看起来更紧迫,旧 P0 永远是"下次"。
**对策**:
- 下次开工先 Read `下次深度优化方向_*.md` 最新版,优先清旧 P0
- 如果用户给的新需求和旧 P0 冲突,**主动指出 trade-off** 让用户选,不要默认接新需求

### 5.2 SKILL 自己没按 SKILL 写

写 SKILL 的过程没走 SKILL 第 7 节的五步:
- ✅ 侦察 (WebSearch + WebFetch)
- ❌ 设计 (没列章节大纲 review,直接写)
- ✅ 编码 (Write + Edit)
- ❌ 测试 (无,SKILL 怎么"测"是开放问题 — 看看 markdown 渲染?让 LLM 跑一遍 review checklist 自检?)
- ❌ 复盘 (本节算补做的)

**对策**: SKILL 自己也要按自己的工作流写。下次 v1.2 必须有"设计"和"复盘"环节痕迹。

### 5.3 备份目录命名混乱

今天产了 `工作脚本备份_20260514_CLI工具/`,本次又要再加一个 SKILL 体系的。两个备份内容会有重叠。
**对策**: 改用单备份目录 `工作脚本备份_20260514/`,内部分子目录 `cli/` `skill/` `docs/`。下次开工把今天的两个目录合并。
